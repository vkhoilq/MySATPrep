# src/lib/questionbank/

## Responsibility
Complete question-browser module responsible for fetching, filtering, sorting, and progressively rendering question results in the Question Bank view. Implements the full state management (via `useReducer` + `useEffect`), data fetching pipeline (batched + debounced), filter pipeline (difficulty, skill, Bluebook exclusion/inclusion, date range, answer status), infinite-scroll pagination via `IntersectionObserver`, and reusable render helpers. This is the data engine behind the question bank UI.

## Design
- **State machine via useReducer** — `useQuestionResults.ts` drives a `QuestionResultsState` through a `questionResultsReducer` (in `reducer.ts`) with 19 action types. State includes: `questionsWithData[]`, `fetchedQuestionIds` (Set to prevent duplicate fetches), `visibleCount` (pagination cursor), `isLoadingMore`, filter selections (`selectedDifficulties`, `selectedSkills`, `excludeBluebookQuestions`, `onlyBluebookQuestions`), `sortOrder`, `dateRange`, and `answerStatus`. Filter changes reset `visibleCount` to `INITIAL_VISIBLE_COUNT` (10) and clear `isLoadingMore`.
- **Composable filter pipeline** — `filters.ts` provides pure functions composable in order: `sortQuestionsByDate` → `filterQuestionsByDateRange` → `filterOutBluebookQuestions`/`filterOnlyBluebookQuestions` → `filterQuestionsByDifficulty` → `filterQuestionsBySkills` → `filterQuestionsByAnswerStatus`. Each function accepts `QuestionWithData[]` and returns a filtered copy. The combined `filterQuestions()` function orchestrates the full pipeline. A simpler `filterQuestionsBasic()` skips Bluebook and sort/date/status filters.
- **Progressive fetching with batching** — `useQuestionResults` debounces (100ms) then incrementally fetches question detail data for visible questions only. Questions are processed in batches of `API_BATCH_SIZE` (3) with `API_BATCH_DELAY` (200ms) between batches. Results are dispatched as `SET_QUESTION_SUCCESS` or `SET_QUESTION_ERROR`. Fetched IDs are tracked in a Set to avoid re-fetching.
- **Infinite scroll** — An `IntersectionObserver` watches a sentinel `div` ref (`loadMoreRef`). When it intersects and `hasMoreQuestions` is true, it dispatches `INCREASE_VISIBLE_COUNT` by `LOAD_MORE_BATCH_SIZE` (10) after a `LOAD_MORE_DELAY` (300ms). Configuration constants: `OBSERVER_THRESHOLD: 0.1`, `OBSERVER_ROOT_MARGIN: "100px"`.
- **Custom hooks for derived UI data** — `hooks.ts` provides `useLoadingIndicatorData` (loading counts, missing-difficulty notes), `useSkillOptions` (groups skills by `primaryClassCd` for grouped select dropdowns), `useRenderSelectedDifficulties` and `useRenderSelectedSkills` (human-readable labels for multi-select). `components.tsx` provides `LoadingIndicator` render component and `useRenderDifficultyOption`/`useRenderSkillOption` callbacks.
- **API abstraction** — `api.ts` exports `fetchQuestionData(questionId)` calling `/api/question-by-id/{id}` and returning `QuestionById_Data | null`.
- **Barrel export** — `index.ts` re-exports all types, constants, filters, reducer, API, hooks, and components.

## Flow
1. **Initialization** — `useQuestionResults(questions, selectedSubject, bluebookExternalIds)` receives a `PlainQuestionType[]` array. On mount, it dispatches `INITIALIZE_QUESTIONS` wrapping each question with `{ isLoading: true, hasError: false, timestamp }` into `QuestionWithData[]`. `RESET_FETCHED_IDS` clears the fetch-dedup set.
2. **Filtering** — `actualFilteredQuestions` is a `useMemo` computed by `filterQuestions()` using the current state filters. This produces the viewport slice. Changes to any filter or sort dispatch resets `visibleCount` to 10.
3. **Fetching** — A debounced effect (100ms) runs when `state.isInitialized`, `actualFilteredQuestions`, or `state.visibleCount` changes. It identifies visible questions (slice 0→visibleCount) that are still `isLoading` with no `questionData` and not already in `fetchedQuestionIds`. It dispatches `ADD_FETCHED_ID` for each, then processes batches of 3 concurrently with 200ms inter-batch delay. Success dispatches `SET_QUESTION_SUCCESS` with the `QuestionById_Data`, error dispatches `SET_QUESTION_ERROR` with a message.
4. **Pagination** — The `IntersectionObserver` effect watches the sentinel div. When scrolled into view and `hasMoreQuestions` is true (visibleCount < filtered length), it dispatches `SET_LOADING_MORE` then after 300ms dispatches `INCREASE_VISIBLE_COUNT` (+10) and clears loading.
5. **Retry** — `handleRetry(index, questionId)` finds the question in state array, dispatches `REMOVE_FETCHED_ID` (to allow re-fetch) then `SET_QUESTION_LOADING`.
6. **Filter interactions** — `SET_DIFFICULTY_FILTER`/`SET_SKILL_FILTER` replace the full selection array and reset visibleCount. `TOGGLE_EXCLUDE_BLUEBOOK` and `TOGGLE_ONLY_BLUEBOOK` are mutually exclusive (setting one to `true` forces the other to `false`). `SET_SORT_ORDER`, `SET_DATE_RANGE`, `SET_ANSWER_STATUS` each reset visibleCount.

## Integration
- **Consumed by**: `src/app/` page/layout components that render the question bank UI (likely a page with assessment/subject/domain selection that passes `PlainQuestionType[]` and `bluebookExternalIds` into this hook), and filter UI components that dispatch actions through the returned `dispatch` function
- **Depends on**: `@/types/question` (PlainQuestionType, QuestionById_Data, QuestionDifficulty), `@/types/statistics` (AnsweredQuestion), `@/components/ui/calendar` (RangeValue), `src/lib/questionbank/api.ts` (fetchQuestionData), React (`useEffect`, `useCallback`, `useMemo`, `useReducer`, `useRef`), browser APIs (`IntersectionObserver`, `AbortSignal`)
